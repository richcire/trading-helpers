# 🎛️ Trading Helper Design System Guide

> 이 문서는 트레이딩 헬퍼의 공통 디자인 시스템 단일 소스다.
> 앞으로 UI를 구현하는 에이전트는 **반드시 이 문서를 먼저 읽고**, 그 다음 `src/components/agents.md`, `src/features/agents.md`를 읽는다.
> 새 화면이나 컴포넌트가 이 문서의 규칙을 벗어나면, 먼저 디자인 시스템을 확장한 뒤 코드를 작성한다.

---

## 0. 목적

이 문서의 목적은 다음 3가지다.

1. 모든 기능 화면이 하나의 제품처럼 보이게 한다.
2. 에이전트가 새 UI를 만들 때 스타일 결정을 다시 하지 않게 한다.
3. 기능 구현보다 먼저 `공통 토큰`, `공통 패턴`, `상태 규칙`을 강제한다.

이 서비스는 일반 랜딩 페이지가 아니라 **트레이딩 계산 워크스페이스**다.
따라서 기준은 “예쁜 UI”보다 아래에 둔다.

- 입력 속도
- 숫자 가독성
- 상태 판독성
- 모바일 대응
- 반복 가능한 컴포넌트 구조

---

## 1. 제품 디자인 방향

### 1.1 핵심 성격

- 제품 성격: `트레이딩 데스크`
- 인상: `정밀함`, `차분함`, `계산 도구`, `전문성`
- 금지: 감성형 랜딩 페이지 스타일, 과한 글로우, 장식성 우선 레이아웃

### 1.2 기본 원칙

- 다크 퍼스트 단일 테마를 사용한다.
- 입력과 결과는 같은 시각 언어를 공유해야 한다.
- 상승/하락은 장식색이 아니라 **의미색**이다.
- 여백은 충분해야 하지만 정보 밀도를 해치면 안 된다.
- 카드와 패널은 “정밀 패널”처럼 보여야 하며, 둥글고 말랑한 소비자 앱처럼 보이면 안 된다.

### 1.3 우선순위

UI 결정을 할 때 아래 우선순위를 따른다.

1. 계산 흐름이 빠른가
2. 숫자와 상태가 즉시 읽히는가
3. 모바일에서도 무너지지 않는가
4. 공통 컴포넌트로 재사용 가능한가
5. 그 다음이 미적 완성도다

---

## 2. 에이전트 작업 규칙

### 2.1 반드시 지킬 것

- 기존 토큰과 공통 컴포넌트를 우선 사용한다.
- 새 스타일이 필요하면 먼저 이 문서에 맞는 `공통 패턴`인지 판단한다.
- 같은 역할의 UI를 기능마다 다른 방식으로 만들지 않는다.
- 컴포넌트 내부에서 임의 색상값을 반복 선언하지 않는다.
- 손익 상태 색상은 반드시 디자인 시스템의 의미색을 사용한다.

### 2.2 먼저 검토할 파일

새 UI 작업 전 확인 순서:

1. `design_system.md`
2. `src/components/agents.md`
3. 관련 feature의 `agents.md`
4. 현재 존재하는 공통 컴포넌트 구현

### 2.3 새 컴포넌트 추가 기준

아래 둘 중 하나면 공통 컴포넌트로 승격한다.

- 두 개 이상의 기능에서 재사용될 가능성이 높다
- 상태 규칙이 반복된다

반대로 아래면 feature 내부 로컬 컴포넌트로 둔다.

- 특정 계산기 한 곳에서만 쓰는 조합 UI다
- 비즈니스 맥락이 강해서 범용화 이득이 없다

## 3. 토큰 시스템

### 3.1 단일 소스

디자인 토큰의 단일 소스는 `src/index.css`다.

- 색상
- 그림자
- 반경
- spacing scale
- 컨테이너 폭
- transition preset

에이전트는 새 전역 토큰이 필요할 때 먼저 `src/index.css`를 수정한다.

### 3.2 현재 핵심 색상 토큰

반드시 아래 의미를 유지한다.

| 토큰 | 역할 |
|------|------|
| `--color-bg-app` | 앱 전체 배경 |
| `--color-bg-surface` | 기본 카드/패널 배경 |
| `--color-bg-elevated` | 더 강조된 표면, 모달/강조 카드 |
| `--color-border-subtle` | 기본 보더 |
| `--color-border-strong` | 강조 보더 |
| `--color-text-primary` | 핵심 텍스트 |
| `--color-text-secondary` | 보조 텍스트 |
| `--color-text-muted` | 힌트/설명/비활성 톤 |
| `--color-accent` | 활성 상태, 포커스, 선택 강조 |
| `--color-accent-soft` | 활성 tint 배경 |
| `--color-profit` | 수익/상승 상태 |
| `--color-loss` | 손실/위험 상태 |
| `--color-warning` | 경고/주의 상태 |
| `--color-info` | 정보성 강조 |
| `--color-focus-ring` | 키보드 포커스 링 |

### 3.3 의미색 규칙

- `profit`: 수익, 익절, 양의 기대값, 성공 상태
- `loss`: 손실, 손절, 검증 실패, 위험 액션
- `warning`: 현재가, 주의, reset류 액션
- `accent`: 활성 탭, 선택된 세그먼트, 주요 포커스 상태
- `neutral`: 기본 상태, 일반 카드, 일반 텍스트

금지:

- 기능마다 다른 초록/빨강 계열을 임의로 정의하는 것
- 의미색을 장식 목적으로 남용하는 것

### 3.4 간격 및 반경

기본 기준:

- spacing scale: `4 / 8 / 12 / 16 / 24 / 32`
- `--radius-panel`: 큰 카드와 모달
- `--radius-control`: 입력, 버튼, 세그먼트, 내부 블록
- `--radius-pill`: 배지, 탭, 캡슐형 요소

새 컴포넌트는 이 반경 체계 밖의 값을 가능한 만들지 않는다.

## 4. 타이포그래피 규칙

### 4.1 공통 원칙

- 외부 폰트는 추가하지 않는다.
- 시스템 폰트 스택을 사용한다.
- 모든 숫자 출력은 가능한 `tabular-nums` 성격을 사용한다.

### 4.2 역할별 규칙

- 앱 타이틀: 세미볼드 이상, 촘촘한 tracking
- 섹션 타이틀: 중간 크기 헤드라인
- 라벨: 작은 크기 + uppercase 또는 높은 구분감
- 힌트/설명: 작은 크기 + secondary/muted
- 핵심 수치: 큰 크기 + `text-data`

### 4.3 숫자 표시 원칙

다음 값들은 반드시 정렬성과 판독성을 유지해야 한다.

- 가격
- 손익
- 퍼센트
- 수량
- ROI
- 기대값

숫자 표시는 기본적으로 아래를 따른다.

- 오른쪽 정렬
- `text-data` 사용
- 포맷 함수(`formatNumber`, `formatPnl`, `formatPct`) 우선 사용

---

## 5. 레이아웃 시스템

### 5.1 앱 셸 구조

기본 구조는 아래를 유지한다.

1. `Header`
2. `TabNav`
3. `main` content container
4. 필요 시 `SettingsModal`

### 5.2 컨테이너 규칙

- 메인 컨테이너는 `max-w-[var(--container-wide)]` 기준
- 내부 기능 패널은 `1열` 또는 `입력/결과 2열` 구조
- 모바일에서는 무조건 1열 우선
- 데스크탑에서만 2열 이상을 사용한다

### 5.3 sticky 요소 규칙

- 헤더는 항상 sticky
- 탭도 sticky
- sticky 요소는 배경 blur 또는 반투명 표면을 사용해 스크롤 중 분리감을 유지한다

### 5.4 배경 규칙

- 전체 앱은 단색 평면 배경을 피한다
- 하지만 장식 그래픽을 과하게 넣지 않는다
- 그라데이션과 tint는 미세해야 하며 데이터 가독성을 해치면 안 된다

---

## 6. 공통 표면 규칙

### 6.1 표면 수준

`SurfaceLevel` 의미:

- `base`: 기본 카드
- `raised`: 조금 더 강조된 카드
- `overlay`: 모달/오버레이 표면

### 6.2 카드 원칙

모든 결과는 가능한 카드 내부에서 보여준다.

카드 기본 구조:

1. eyebrow 또는 title
2. 핵심 설명
3. 주요 값
4. metric rows
5. 보조 액션

카드에 과한 색채를 직접 칠하지 않는다.
상태가 있어도 보통은 아래 순서로만 강조한다.

1. 숫자 색상
2. 배지
3. 미세한 tint
4. 필요하면 ring

### 6.3 그림자 규칙

- 그림자는 입체감보다 표면 분리를 위한 용도다
- 무겁고 흐린 그림자 남용 금지
- `surface`와 `modal` 두 단계만 유지

---

## 7. 공통 컴포넌트 규칙

### 7.1 기본 원칙

- 공통 컴포넌트는 모두 named export 사용
- props 타입은 파일 상단 `interface Props`
- 비즈니스 계산 로직 포함 금지
- 상태는 props로 받고, 계산은 feature 훅 또는 `calc/`에서 처리

### 7.2 현재 공통 컴포넌트 목록

레이아웃:

- `Header`
- `TabNav`
- `SettingsModal`

프리미티브:

- `SectionCard`
- `ActionButton`
- `IconButton`
- `SegmentedControl`
- `MetricRow`
- `StatusDot`
- `FieldLabel`
- `FieldHint`
- `FieldError`

UI:

- `InputRow`
- `ResultCard`
- `PnlBadge`
- `CopyButton`

### 7.3 컴포넌트 확장 원칙

새 기능을 만들 때 아래 순서로 검토한다.

1. 기존 컴포넌트 조합으로 해결 가능한가
2. 기존 공통 컴포넌트 props 확장으로 해결 가능한가
3. 정말 공통 primitive가 더 필요한가
4. 그때만 새 공통 컴포넌트를 추가한다

---

## 8. 입력 시스템 규칙

### 8.1 InputRow 기본 규칙

- 숫자 입력은 우측 정렬
- 단위는 우측 suffix로 고정
- 라벨, 필드, 에러, 힌트를 한 세트로 다룬다
- 높이는 최소 터치 영역을 만족해야 한다

### 8.2 상태 규칙

입력은 아래 상태를 명확히 구분해야 한다.

- default
- focus
- error
- disabled

원칙:

- error는 보더만 빨갛게 만들지 말고 메시지도 바로 아래 보여준다
- disabled는 판독 가능해야 한다
- focus는 accent 계열 링 또는 보더 강화로 표현한다

### 8.3 입력값 처리

입력값은 `string`으로 관리한다.
계산 전에 파싱한다.

금지:

- input value를 상태에서 곧바로 number로 들고 있는 것
- 유효성 검증 없이 계산 함수에 넘기는 것

---

## 9. 결과 시스템 규칙

### 9.1 ResultCard 규칙

- 결과는 카드 형태로 묶는다
- 핵심 값 1개를 가장 크게 보여준다
- 세부 값은 `MetricRow`로 정리한다
- 카드 액션은 우상단 또는 하단 끝 정렬

### 9.2 PnlBadge 규칙

- 양수: `profit`
- 음수: `loss`
- 0: `neutral`

배지는 “상태 요약” 역할이다.
핵심 수치를 대신하는 용도로 남용하지 않는다.

### 9.3 숫자 포맷

- 손익 금액: `formatPnl`
- 퍼센트: `formatPct`
- 일반 숫자: `formatNumber`

기능 구현 시 직접 `toLocaleString()`을 흩뿌리지 않는다.

---

## 10. 액션 시스템 규칙

### 10.1 버튼 계층

버튼은 아래 계층을 따른다.

- `solid`: 가장 중요한 액션
- `outline`: 보조 액션
- `ghost`: 저강도 액션

### 10.2 tone 사용 규칙

- `accent`: 주요 작업
- `neutral`: 일반 작업
- `loss`: 위험 작업
- `warning`: 리셋, 주의 필요 작업
- `profit`: 성공 상태 확인 또는 긍정 액션

한 화면에서 동시에 너무 많은 `solid` 버튼을 만들지 않는다.

---

## 11. 탭과 내비게이션 규칙

### 11.1 TabNav 원칙

- 탭은 단순 버튼이 아니라 워크스페이스 모드 선택기다
- 모바일 가로 스크롤을 허용한다
- 활성 탭은 배경 tint + 보더 + 하단 강조선으로 구분한다
- 비활성 탭은 지나치게 흐리게 만들지 않는다

### 11.2 탭 텍스트 길이

- 긴 한글 라벨도 깨지지 않아야 한다
- 터치 영역 최소 높이를 유지한다

---

## 12. 모달 규칙

### 12.1 SettingsModal 원칙

- 모달은 즉시 반영형 설정 패널이다
- 별도 저장 버튼을 두지 않는다
- reset만 별도 액션으로 둔다

### 12.2 접근성 규칙

- 배경 클릭으로 닫힘
- ESC로 닫힘
- 열릴 때 배경 스크롤 잠금

새 모달도 기본적으로 이 규칙을 따른다.

---

## 13. 차트 스타일 규칙

차트 구현 시 `Recharts`에 아래 규칙을 적용한다.

### 13.1 기본 규칙

- 차트 배경은 투명 또는 surface 계열
- grid와 axis는 muted 처리
- tooltip은 일반 카드와 같은 표면 규칙 사용

### 13.2 reference color 고정

- 평균단가: `neutral`
- 손절가: `loss`
- 익절가: `profit`
- 현재가: `warning`

### 13.3 금지

- 네온 글로우 차트
- 과한 다색 그라데이션
- 축과 보조선이 데이터보다 눈에 띄는 스타일

---

## 14. 상태 표현 규칙

모든 공통 UI는 아래 상태를 구분할 수 있어야 한다.

- default
- hover
- focus-visible
- active
- disabled
- error
- profit
- loss
- warning

원칙:

- 색만으로 상태를 표현하지 않는다
- 필요 시 텍스트, 배지, 아이콘, 보더를 함께 사용한다

---

## 15. 반응형 규칙

### 15.1 모바일 우선

- 375px 폭에서도 깨지지 않아야 한다
- 탭은 가로 스크롤 허용
- 입력과 결과는 1열
- 숫자가 길어도 레이아웃이 튀지 않게 해야 한다

### 15.2 데스크탑 확장

- 넓어진 화면에서는 `입력 패널 / 결과 패널` 2열 허용
- 하지만 지나치게 한 줄 길이가 길어지면 안 된다

---

## 16. 기능 페이지 구현 규칙

각 feature 페이지는 이 디자인 시스템 위에서 아래 패턴을 따른다.

### 16.1 추천 구조

1. 페이지 요약 또는 핵심 설정
2. 입력 카드
3. 결과 카드
4. 보조 시각화 또는 설명

### 16.2 기능 구현 시 우선 재사용할 것

- 방향 선택: `SegmentedControl`
- 입력: `InputRow`
- 결과 섹션: `SectionCard` 또는 `ResultCard`
- 요약 수치: `MetricRow`
- 손익 상태: `PnlBadge`
- 복사 액션: `CopyButton`

### 16.3 feature 내부에서 허용되는 것

- 해당 기능 전용 로컬 row 컴포넌트
- feature 전용 레이아웃 조합
- 특정 계산 결과 전용 텍스트 조합

### 16.4 feature 내부에서 금지되는 것

- 공통 버튼 스타일을 새로 작성하는 것
- 공통 입력 스타일을 새로 작성하는 것
- 손익 색상을 feature마다 다르게 쓰는 것

## 18. 문서 업데이트 규칙

아래 경우 이 문서를 반드시 함께 수정한다.

- 새 공통 컴포넌트 추가
- 전역 토큰 추가/변경
- 상태 규칙 변경
- 차트 스타일 기준 변경

반대로, feature 하나에만 해당하는 UI 설명은 이 문서가 아니라 해당 feature 문서에 적는다.
